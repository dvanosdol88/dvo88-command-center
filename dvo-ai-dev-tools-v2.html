ðŸ”‘ Key Concepts Explained

1. What is "Cursor Agent"?

Cursor Agent is NOT its own AI model â€” it's an orchestration layer that wraps frontier models (Claude, GPT, Gemini, etc.). Cursor built Composer, their own fast coding model, but also supports all major providers. When you use "Agent mode" in Cursor, it autonomously explores your codebase, edits files, and runs commands â€” powered by whichever model you select.

TL;DR: Cursor = orchestration + UI + tools. The AI brain comes from Claude/GPT/Gemini.

Billing: Cursor has its own subscription ($20/mo Pro) with usage limits. API calls flow through Cursor's account, not yours directly.

2. Config Files: One or Many?

Each agent has MULTIPLE config files for different purposes:

Claude Desktop:
â€¢ settings.json â€” MCP servers, permissions
â€¢ CLAUDE.md â€” instructions/rules
â€¢ plugins/*/plugin.json â€” per-plugin config

Gemini CLI:
â€¢ extensions/ â€” built-in extensions
â€¢ antigravity/mcp_config.json â€” custom MCP servers
â€¢ GEMINI.md â€” instructions

Codex CLI: config.toml + mcp_servers.json + AGENTS.md
Cursor: mcp.json + .cursorrules

3. What's the difference between (config) and (plugin)?

(config) = MCP server defined in a JSON config file (mcp_config.json, mcp.json). You manually specify the command and args. Can be enabled/disabled.

(plugin) = A higher-level package with metadata, skills, and hooks. Plugins have a plugin.json describing the tool, plus optional .mcp.json for the server. Plugins can include multiple capabilities beyond just an MCP server.

Example: Vercel in Claude Desktop is a plugin (has /deploy skill). Playwright in Gemini's mcp_config.json is a config.

4. How hard is "VS Code auto-sync"?

Difficulty: Easy-Medium (30-60 min setup)

Option A: VS Code Task on save â€” when you save a .md file, run a Node script that regenerates this HTML. Uses tasks.json + a simple script.

Option B: File watcher with chokidar â€” runs in background, auto-rebuilds on change.

Option C: Pre-commit hook â€” rebuilds HTML before any git commit.

All are doable in an afternoon. Option A is simplest.

Windows OpenSSH Gotchas

1. No file extensions!
Must be authorized_keys not .txt

2. Admin users use different key location
Standard: ~\.ssh\authorized_keys
Admin: C:\ProgramData\ssh\administrators_authorized_keys

3. Permissions must be locked down
icacls ... /inheritance:r /grant "Administrators:F" /grant "SYSTEM:F"

4. Config: C:\ProgramData\ssh\sshd_config

Ubuntu SSH Setup (ThinkBox)

Install: sudo apt install openssh-server
Enable: sudo systemctl enable ssh && sudo systemctl start ssh
Get IP: hostname -I

Key auth:
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
